We're excited to announce the release of a new way to manage access to your assessments! It introduces a simpler mental model and a full-featured in-browser editor. At the same time, the old `allowAccess` system remains fully supported, so there's no rush to migrate.

## Edit access settings in the UI

Configuring access for an assessment no longer requires hand-crafting complex data structures in JSON. All access control settings can now be configured directly in your browser with a powerful editor. Release dates, due dates, early and late deadlines, time limits, passwords, PrairieTest links, review visibility, and student-specific exceptions all live in one place.

<BlogImage src={editor} alt="The in-browser editor for access control" full />

## Dates, deadlines, and credit

The biggest change is that assessment access is now modeled as a single timeline. Instead of writing overlapping `allowAccess` rules and reasoning through which one applies, you configure a release date, an optional due date, optional early and late deadlines, and what should happen after the last deadline.

For example, a homework can be worth 110% until an early deadline, 100% until the due date, 80% for a late-submission window, and then remain open for practice at 0%. The editor presents that as one ordered set of deadlines, making it much easier to see what students will experience.

## Time limits, passwords, and review settings

The same editor also handles settings that used to be scattered across access rules: time limits, passwords, and what students can see after an assessment is complete.

For timed assessments, you can give each student a fixed amount of working time after they begin. You can also require a password before students start or continue working. After the assessment is complete, you can control when students can see questions and scores, including optionally revealing them on a future date.

## PrairieTest exams

The UI makes it easy to link PrairieLearn assessments with PrairieTest exams. When a student is in an active PrairieTest reservation, PrairieTest controls the access window and time limit.

- An exam link can now be marked as **read-only**. This is useful for courses that want to give students an opportunity to review their work in a controlled setting. This replaces the old `active` setting.
- It's now easier to configure what students see during their reservation after they've completed their exam. By default, students will see all their work and scores. When real-time grading is disabled, instructors can configure the exam to either show the overall scores but hide the questions, or hide both the scores and questions. This replaces the old `showClosedAssessment` and `showClosedAssessmentScore` settings, and makes common use cases easier to configure.

## Overrides for student labels and individual students

Assessment access will often need exceptions:

- Students with accessibility accommodations will need extra time.
- Different sections might need different deadlines.
- A student might need a deadline extension due to illness.

In the new access control system, these are represented as **Overrides**. An override applies to either individual students or student labels and "overrides" specific settings. Here's how they can be used to support the above scenarios:

- If the default time limit is 60 minutes, an override for a "150% time" student label can increase the time limit to 90 minutes.
- An override for the "Section A" label can set a due date of June 10th, and an override for "Section B" can set a due date of June 12th.
- If the default due date is July 10th, an override for "Matthew West" can set the due date to July 15th.

Overrides stack, and overrides for individual students take precedence over overrides for specific student labels. So, if Matthew West had the "150% time" and "Section B" labels, he would have 90 minutes to take the exam, and would have a due date of July 15th.

Overrides for student labels are stored under `accessControl` in `infoAssessment.json`, which makes them easy to review and reuse across course copies. Overrides for individual students are stored in the database instead, so one-off accommodations don't become part of the course repository.

## Migrating to the new access control system

While legacy `allowAccess` rules continue to work, we think most instructors will find the new system significantly easier to use. When you're ready to switch, you have a few options:

- When copying a course instance, PrairieLearn will prompt you to convert any legacy rules to the new format.
- You can migrate individual assessments from their "Access" tab by clicking "Migrate to modern format". You can preview the resulting configuration and review any warnings or incompatibilities before confirming.

Note that because of the extreme flexibility of the old system, there are some niche access patterns that can't be expressed directly in the new one. PrairieLearn will warn you when this is the case.

Access rules using the `uids` array don't have a direct JSON equivalent, as student-specific rules aren't stored in JSON. After migrating, you can manually add appropriate overrides for specific students or student labels.

## JSON is still there when you need it

The new UI on the "Access" tab makes it easy to configure every aspect of access for your assessment. However, the file-backed parts of the configuration are also controllable via the new `accessControl` property in `infoAssessment.json` files. Student-label overrides live there too, while student-specific overrides are managed through the UI. This can be used for advanced use cases like programmatic creation or editing of assessments. The JSON format is documented in the [access control documentation](https://docs.prairielearn.com/assessment/accessControlModern).
